## 1. 개요

앱 상단에 **진행률 대시보드**를 두고, 할 일 **인라인 수정**(더블클릭 편집)과 **UI 애니메이션**(진행률 바·목록 추가/삭제·완료 체크)을 적용하였다. 대시보드 수치는 목록 변경 시마다 **같은 렌더 사이클**에서 갱신되며, 삭제 애니메이션 중에도 통계가 지연되지 않도록 **통계 전용 제외 목록**을 둔다.

---

## 2. 진행률 대시보드

### 2.1 배치 및 HTML

| 요소 | 설명 |
|------|------|
| 위치 | `header.app__header` 아래, 카테고리 필터(`#task-filters`) 위 |
| 루트 | `section#dashboard.dashboard` (`aria-label="진행률 대시보드"`) |
| 전체 요약 | `#dashboard-summary` — 예: `5/10 완료 (50%)` (항목이 없으면 안내 문구) |
| 전체 바 | `#dashboard-progress`(역할 `progressbar`) + 내부 `#dashboard-bar-fill` |
| 카테고리 | `#dashboard-categories` — 행마다 라벨·`완료/전체` 수치·미니 진행률 트랙 |
| 오늘 추가 | `#dashboard-today` — 로컬 달력 기준 **오늘**에 `createdAt`이 속한 항목 개수 |

### 2.2 계산 규칙

- **전체:** `statTasks` 기준 총 개수·완료 개수·백분율(반올림). `statTasks`는 아래 2.4와 같이 정의된다.
- **카테고리별:** `work` / `personal` / `study` 각각에 대해 동일하게 완료 수·전체 수·미니 바 너비(%) 산출.
- **오늘 추가:** `createdAt`을 타임스탬프로 변환하여 당일 0시~23:59:59.999 구간에 포함되는지 판별.

### 2.3 실시간 갱신

`render()` 진입 시 항상 `updateDashboard()`를 먼저 호출한다. 따라서 추가·삭제·완료 토글·인라인 저장·필터 전환 등 `render()`가 호출되는 모든 경로에서 대시보드가 목록 상태와 일치한다.

### 2.4 삭제 중 통계 (`pendingRemoveIds`)

삭제 시 행에 페이드·슬라이드 전환을 주는 동안 DOM에는 항목이 남아 있으나, 사용자에게는 이미 “삭제된 것”처럼 보이도록 **통계에서만** 해당 `id`를 제외한다.

- 애니메이션 시작 시 `pendingRemoveIds[id] = true` 후 `updateDashboard()`만 반영(목록 전체 재그리기 없이 수치만 갱신).
- 애니메이션 종료 시 `delete pendingRemoveIds[id]` 후 실제 배열에서 제거·저장·`render()`.

## 3. 인라인 수정

| 동작 | 설명 |
|------|------|
| 진입 | **미완료** 할 일의 본문 텍스트(`.task-item__text`) **더블클릭** |
| UI | 텍스트·태그 대신 `input.task-item__edit-input` + `select.task-item__edit-select` |
| 저장 | **Enter**(입력란·select 모두). 빈 문자열( trim 후 )이면 저장하지 않고 편집 종료에 준함 |
| 취소 | **ESC** |
| 완료된 항목 | 더블클릭 편집 비활성(체크된 할 일은 읽기 전용) |
| 편집 중 완료 체크 | 체크박스 `disabled`로 막음 |
| 필터 변경 | 필터 버튼 클릭 시 `editingTaskId` 초기화 후 재렌더 |
| 삭제 | 편집 중이어도 삭제 가능(삭제 클릭 시 편집 상태 해제 후 삭제 플로우) |

데이터 반영은 `updateTask(id, { text, category })`로 처리하며, 저장 후 `saveTasks()` 및 `render()`를 수행한다.

## 4. UI 애니메이션

| 구분 | 구현 요약 |
|------|-----------|
| 진행률 바 | `.dashboard__bar-fill`, `.dashboard__cat-bar-fill`에 `width` 전환(`--transition-slow`) |
| 항목 추가 | 새 항목 id에 `lastAddedTaskId` 부여 → 해당 `li`에 `task-item--enter` + `@keyframes taskFadeIn` |
| 항목 삭제 | `task-item--removing`으로 `opacity`·`transform` 전환 후 DOM·데이터 제거 |
| 완료 체크 | 완료로 바뀔 때 `completeSlideTaskId` 설정 → `task-item--complete-slide`로 본문 `taskCompleteSlide` 애니메이션 |
| 접근성 | `prefers-reduced-motion: reduce`일 때 전환·키프레임 애니메이션 완화 |

## 5. 데이터·저장소와의 관계

- 할 일 객체 스키마는 기존과 동일하게 `id`, `text`, `completed`, `createdAt`, `category`를 사용한다(`02` 보고서의 v2 정책과 호환).
- 대시보드·애니메이션용 상태(`editingTaskId`, `lastAddedTaskId`, `completeSlideTaskId`, `pendingRemoveIds`)는 **메모리 전용**이며 `localStorage`에 쓰이지 않는다.

## 7. 확인 권장 사항

1. 할 일을 여러 개 추가·완료·삭제하면서 상단 **전체/카테고리/오늘 추가** 수치가 즉시 맞는지 본다.  
2. 삭제 직후 통계가 바로 줄고, 행은 잠시 페이드되는지 확인한다.  
3. 미완료 항목 텍스트를 더블클릭해 내용·카테고리를 바꾼 뒤 Enter 저장·ESC 취소를 각각 시험한다.  
4. OS 설정에서 “애니메이션 줄이기”를 켠 브라우저에서 과한 움직임이 줄었는지 본다.

## 8. 결론

진행률 대시보드(전체·카테고리·오늘 추가), 인라인 수정(더블클릭·Enter/ESC·카테고리 select), 진행률·목록·완료 관련 UI 전환을 구현하였고, `render()`와 `pendingRemoveIds`를 통해 **대시보드가 목록 조작과 실시간에 가깝게 동기화**되도록 정리하였다.
